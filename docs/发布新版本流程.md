# 🚀 新版本发布流程（固定文件名方案）

> **适用场景**：每次发布 GJJ-OCR-Tool 新版本时的操作步骤
> **方案**：安装包使用固定文件名 `GJJ-OCR-Tool-latest-setup.exe`，每次发布覆盖上传，无需修改 URL

---

## 📋 前置条件

1. ✅ 私钥文件 `apps/desktop/tauri.key` 存在（**不能丢失！丢失后无法签名更新包**）
2. ✅ 公钥已配置在 `apps/desktop/src-tauri/tauri.conf.json` 中
3. ✅ 已配置好 CDN/服务器地址（`version.json` 上传位置）
4. ✅ 当前代码已完成测试，无已知问题

---

## 步骤 1：修改版本号（4 个文件）

将 `1.2.0` 改为新版本号（如 `1.2.1`）：

### 1.1 `apps/desktop/package.json`
```json
{
  "version": "1.2.1",
  ...
}
```

### 1.2 `apps/desktop/src-tauri/Cargo.toml`
```toml
[package]
version = "1.2.1"
```

### 1.3 `apps/desktop/src-tauri/tauri.conf.json`
```json
{
  "version": "1.2.1",
  ...
}
```

### 1.4 `apps/desktop/src/App.tsx`
```typescript
const APP_VERSION = "1.2.1";
```

---

## 步骤 2：编写更新日志

修改 `apps/desktop/src/changelog.json`，添加新版本记录：

```json
{
  "versions": [
    {
      "version": "1.2.1",
      "date": "2025-01-20",
      "notes": [
        "修复 xxx 问题",
        "优化 xxx 功能"
      ]
    },
    ...
  ]
}
```

---

## 步骤 3：打包安装程序

### 3.1 构建前端
```bash
cd apps/desktop
npm run build
```

### 3.2 构建 Tauri 安装包
```bash
cd apps/desktop
npm run tauri:build
```

打包完成后，产物在：
```
apps/desktop/src-tauri/target/release/bundle/nsis/
├── GJJ-OCR-Tool_1.2.1_x64-setup.exe   ← 这是安装包
└── ...
```

> **注意**：打包需要 Python 环境正常、后端服务 `gjj-ocr-server.exe` 已存在。

---

## 步骤 4：生成签名

使用私钥对安装包进行签名：

```bash
cd apps/desktop/src-tauri

npx @tauri-apps/cli signer sign \
  --private-key ../../tauri.key \
  --file ../../src-tauri/target/release/bundle/nsis/GJJ-OCR-Tool_1.2.1_x64-setup.exe
```

执行后会输出类似：
```
Signature: dW50cnVzdGVkIGNvbW1lbnQ6IHNpZ25hdHVyZSBmcm9tIHRhdXJpIGtleQp...
```

**复制这个签名内容**（从 `dW50...` 开始到结束），下一步要用。

---

## 步骤 5：创建 version.json

**使用固定文件名方案**，`version.json` 的 URL 永远不用改！

在 `version.json.template` 基础上修改：

```json
{
  "version": "1.2.1",
  "notes": "修复 xxx 问题，优化 xxx 功能",
  "pub_date": "2025-01-20T12:00:00Z",
  "platforms": {
    "windows-x86_64": {
      "signature": "粘贴上一步复制的签名内容",
      "url": "https://你的CDN地址.com/GJJ-OCR-Tool-latest-setup.exe"
    }
  }
}
```

**关键点：**
- `version` 必须与步骤 1 修改的版本号一致
- `signature` 必须粘贴完整，不能有多余空格或换行
- **`url` 永远不用改**，每次都固定指向 `GJJ-OCR-Tool-latest-setup.exe`

---

## 步骤 6：上传文件到 CDN

将以下两个文件上传到你的 CDN/服务器：

| 文件 | 上传路径 | 操作 |
|------|---------|------|
| `version.json` | `https://你的CDN地址.com/version.json` | **覆盖上传** |
| `GJJ-OCR-Tool_1.2.1_x64-setup.exe` | `https://你的CDN地址.com/GJJ-OCR-Tool-latest-setup.exe` | **覆盖上传**（改名为 `GJJ-OCR-Tool-latest-setup.exe`） |

> **CDN 目录结构示例**：
> ```
> 你的CDN/
> ├── version.json                    ← 每次覆盖
> └── GJJ-OCR-Tool-latest-setup.exe   ← 每次覆盖（始终固定文件名）
> ```

---

## 步骤 7：验证更新

1. 打开已安装的旧版本应用（如 v1.2.0）
2. 点击右上角「检查更新」按钮
3. 应该弹出「发现新版本」窗口
4. 点击「立即更新」，应开始下载并自动重启

如果失败，检查：
- `version.json` 是否能通过浏览器直接访问（无鉴权）
- 签名是否正确（有无多余空格）
- 版本号是否一致

---

## ⚠️ 重要注意事项

### 私钥安全
- `tauri.key` **绝对不能上传到 Git！**
- 已加入 `.gitignore`，提交代码时不会包含
- 建议备份到安全位置（如加密 U 盘）

### 版本号一致性
4 个文件 + `version.json` 中的版本号必须一致：
```
package.json         → 1.2.1
Cargo.toml           → 1.2.1
tauri.conf.json      → 1.2.1
App.tsx              → 1.2.1
version.json         → 1.2.1
```

### 签名错误处理
如果签名验证失败：
1. 重新执行步骤 4 生成签名
2. 确保复制完整，不要截断
3. 检查私钥是否正确（是否匹配公钥）

### CDN 要求
- 必须支持 HTTPS
- `version.json` 必须返回 `Content-Type: application/json`
- 不能有 CORS 限制（或允许你的应用域名访问）

## 📌 固定文件名方案的优势

| 对比项 | 固定文件名方案 | 版本号文件名方案 |
|--------|---------------|-----------------|
| 每次发布改 URL | ❌ 不用改 | ✅ 需要改 |
| CDN 文件数量 | 2 个（固定） | 不断增加 |
| 历史版本回退 | 需手动管理 | 自动保留 |
| 适合场景 | 简单快速发布 | 需要多版本共存 |

**本方案适合**：快速迭代、不需要保留历史版本的场景
